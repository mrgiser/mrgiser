Effective Java读书笔记

发表于 2016-08-13 | 分类于 java |

创建和销毁对象

考虑使用静态工厂代替构造函数

  1. 静态工厂具有名称，可读性抢
  2. 不必在每次调用时都创建新对象，单例模式
  3. 可以返回原类型的子类型对象
  4. 在创建类似List Map时，代码简洁
  5. 缺点：类如果不含有公有或者受保护的构造器，就不能被子类化；其与静态函数没有区别。

遇到多个构造函数参数时，考虑使用构建器

  1. 参数多时并可选，构造函数很多，静态工厂也是如此
  2. 构建器，set方法返回this，调用可以连接起来。

使用私有构造器或者枚举类型强化单例属性

  1. 使用反射机制，可以调用私有构造器
  2. 序列化时，提供readResolve方法保证单一
  3. 单元素的枚举类型成为实现单例的最佳方法

避免创建不必要的对象

  1. 不需要new String，重用不可变的String对象
  2. 注意基本类型与装箱类型，优先使用基本类型，装箱类型每次都会新建。
  3. 只有在对象非常重量级时，才考虑使用对象池

消除过期的对象引用

  1. 如果一个栈先增长，再收缩；从栈中弹出来的对象将不会被当做垃圾回收。Object[]
  2. 内存泄露常见来源：缓存、监听器、其他回调

避免使用终结方法

  1. finalizer通常是不可预测的，也是很危险的，一般情况不适用
  2. 中介方法的缺点在于不能保证会被及时的执行
  3. java不保证终结方法会被及时的执行，而且根本不保证他们会被执行
  4. 异常发生在终结方法中，警告都不会打出
  5. 终结方法有一个非常严重的性能损耗
  6. 使用终结方法，记得使用super.finalizer
  7. 使用try finally代替

所有对象都通用的方法

覆盖equals时，遵守通用约定

  1. 类具有自己特有的逻辑相等概念时，覆盖equals
  2. 需要满足自反性、对称性、传递性、一致性，null必不相等
  3. 覆盖equals时必须覆盖hashCode
  4. 不要企图让equals过于智能，加上@Override

覆盖Equals时要覆盖hashCode

  1. equals所用信息不变，code多次执行结果不变
  2. equals相等，code相同
  3. equals不同，code可能相同，散列的分散性
  4. 不要试图从散列码中排除一个关键部分来提高性能
  5. 散列码缓存在对象内部
  6. 延迟初始化散列码

始终要覆盖toString

  1. toString应该包含对象中包含的所有值得关注的信息

谨慎地覆盖clone

  1. final与clone冲突
  2. 对于一个为了继承而设计的类，如果未能提供行为良好的受保护的clone方法，它的子类就不可能实现cloneable接口

考虑实现Comparable接口

  1. 对象小于、等于、大于指定对象时，返回负数、0、正数
  2. 自反性、对称性、传递性

类和接口

使类和成员的可访问性最小化

  1. 信息隐藏与封装是软件设计的原则
  2. 有效的解除组成系统的个模块之间的耦合关系
  3. 尽可能的使每个类或成员不被外界访问
  4. 私有->包级私有->收保护->公有
  5. 受保护的成员应该尽少使用，是对外API的一部分，需要维护
  6. 长度非零的数组总是可变，类具有公有的静态final数组，或者返回这域的访问方法，几乎总是错误的

在公有类中使用set、get而不是公有域

使可变性最小化

  1. 不可变对象比较简单
  2. 线程安全、不要求同步
  3. 缺点：不同的值都需要一个单独的对象

复合优先与继承

  1. 在包内继承，是非常安全的
  2. 进行夸包边界的继承，是非常危险的
  3. 继承打破了封装性
  4. 只有子类真正是超累的子类型时 is-a，才试用继承

接口优与抽象类

  1. java单继承，抽象类受到极大的限制

函数指针实现策略模式 Arrays.sort

优先使用静态成员类

  1. 静态成员类、非静态成员类、匿名类、局部类
  2. 如果成员类不需要访问外围类实例，就使用静态成员类
